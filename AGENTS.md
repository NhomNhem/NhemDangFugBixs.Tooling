# Repository Guidelines

## Project Structure & Module Organization
This repository is a Unity UPM package with source code under `Source~/` and compiled package assets in `Runtime/` and `Analyzers/`. The main projects live in:
- `Source~/DangFugBixs.Generators~/`
- `Source~/DangFugBixs.Analyzers~/`
- `Source~/DangFugBixs.Runtime~/`
- `Source~/DangFugBixs.Common~/`

Tests are colocated with their owning project, for example `Source~/DangFugBixs.Analyzers~/DangFugBixs.Analyzers.Tests/`. Unity-facing assets and editor helpers live in `Editor/`, and example scenes or migration assets live in `GameFeelUnity-Test-Files/`.

## Build, Test, and Development Commands
Use the solution in `Source~/NhemDangFugBixs.Tooling.sln` for local development.

- `dotnet restore` - restore all project dependencies.
- `dotnet build Source~/NhemDangFugBixs.Tooling.sln` - build the full tooling solution from the repo root.
- `dotnet test Source~/DangFugBixs.Analyzers~/DangFugBixs.Analyzers.Tests/DangFugBixs.Analyzers.Tests.csproj` - run analyzer tests.

If you need Unity deployment during build, set `NHEM_UNITY_PROJECT_ROOT` or use `Source~/LocalBuild.props` as described in the repo docs.

## Coding Style & Naming Conventions
Use C# with nullable enabled and standard brace-on-same-line formatting. Keep namespaces under `NhemDangFugBixs.*`. Prefer descriptive analyzer IDs like `ND005` and keep test names explicit, such as `ManualRegistration_OfAutoRegisteredType_ReportsError`.

Follow the existing repo conventions:
- private fields use `_camelCase`
- generated files should include `// <auto-generated />`
- keep analyzer checks symbol-based, not name-only, when possible

## Testing Guidelines
Analyzer tests use xUnit and Roslyn testing helpers in `DangFugBixs.Analyzers.Tests`; do not add NUnit unless the project file is updated intentionally. Add tests for both positive diagnostics and false-positive protection when changing analyzer rules. Prefer one behavior per test method and verify exact diagnostic IDs, spans, and arguments.

## Commit & Pull Request Guidelines
Recent commits follow a conventional format like `feat: ...`, `fix: ...`, and `chore: ...`. Keep commit subjects short and scoped. For pull requests, include a brief summary, mention affected modules, and note any validation run (`dotnet test`, Unity build, or analyzer-only verification).

## Agent Notes
Before editing, inspect the relevant project file and nearby tests. Keep changes minimal and aligned with the existing source generator/analyzer patterns in `Source~/`.
